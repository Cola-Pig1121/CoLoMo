# RAG 检索增强生成

## 为什么需要 RAG

LLM 存在三大问题：
1. **幻觉**：生成内容看似合理但实际错误
2. **知识时效性**：训练数据有截止日期，无法覆盖最新信息
3. **领域知识**：特定领域专业知识处理效率低

RAG（Retrieval-Augmented Generation）在生成答案之前，首先从外部大规模文档数据库中检索出相关信息，并将这些信息融入生成过程之中，从而指导和优化语言模型的输出。

---

## RAG 核心流程

### 1. 文档处理
- **加载**：PDF、Markdown、HTML、TXT 等格式文档
- **分块（Chunking）**：
  - 固定长度分块：简单但可能切断语义
  - 语义分块：按段落/句子切分，保持完整性
  - 重叠分块：相邻块有重叠，避免切断跨块语义
- **向量化（Embedding）**：通过 embedding 模型将文本块转化为向量

### 2. 检索
- 将用户 query 向量化
- 在向量数据库中通过 ANN（近似最近邻）检索相关 chunk
- 常用索引：Faiss、Milvus、ChromaDB、Qdrant

### 3. 生成
- 将检索结果作为 context 注入 prompt
- LLM 基于 context 生成答案
- 可加入 reranker 进一步排序结果

**Chunking 策略选择：**
| 策略 | 优点 | 缺点 |
|------|------|------|
| 固定长度 | 简单、实现成本低 | 可能切断语义 |
| 语义分块 | 保持语义完整性 | 实现复杂 |
| 重叠分块 | 保留跨块信息 | 冗余增加 |

---

## RAG 优化技巧

| 优化方向 | 方法 |
|---------|------|
| 检索质量 | 混合检索（向量+关键词）、query 改写、结果 reranking |
| Context 组织 | 删除无关内容、上下文压缩、渐进式检索 |
| 生成控制 | 抑制无关信息、控制输出长度、引用标注 |
| 索引优化 | 分层索引、摘要索引、实体索引 |

---

## 向量数据库对比

| 数据库 | 优势 | 适用场景 |
|--------|------|---------|
| Faiss | 轻量、易用 | 实验、小规模数据 |
| Milvus | 功能丰富、分布式 | 生产环境、大规模 |
| ChromaDB | 简单、嵌入式 | 快速原型 |
| Qdrant | 高性能、混合检索 | 精确检索需求 |

---

## RAG 评估指标

| 指标 | 衡量内容 |
|------|---------|
| 召回率（Recall） | 相关文档是否被检索到 |
| 精确率（Precision） | 检索结果中相关文档的比例 |
| MRR（Mean Reciprocal Rank） | 首个相关结果的位置 |
| NDCG | 综合考虑相关性和排序位置 |
